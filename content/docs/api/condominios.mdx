# Endpoint de Condomínios

O endpoint de **Condomínios** permite consultar informações gerais dos condomínios cadastrados na AuxiAPI.

Ele foi desenvolvido como parte do desafio **StartAPI - Da Lógica à Prática na Criação de API**, com foco em consultas `GET`, uso de parâmetros de rota, filtros por query params, respostas em JSON, tratamento de erros e separação de responsabilidades entre Controller, Service e Repository.

---

## Visão geral

| Informação | Valor |
| --- | --- |
| Recurso | Condomínios |
| Rota base | `/api/condominios` |
| Método principal | `GET` |
| Formato de resposta | JSON |
| Autenticação | JWT Bearer |
| Paginação | Sim |
| Cache | Sim |

---

## Autenticação

Os endpoints de Condomínios são protegidos por autenticação JWT.

Para consumir a API, envie o token no header da requisição:

```http
Authorization: Bearer {token}
```

Caso o token esteja ausente, inválido ou expirado, a API retorna:

```http
401 Unauthorized
```

## Filtros por query params

A listagem de condomínios permite filtros via query params.

### Filtrar por código do condomínio

```http
GET /api/condominios?CodigoDoCondominio=0001
```

### Filtrar por CNPJ

```http
GET /api/condominios?CNPJDoCondominio=12345678000101
```

### Filtrar por nome

```http
GET /api/condominios?NomeDoCondominio=Residencial
```

### Paginação

```http
GET /api/condominios?Pagina=2
```

### Combinação de filtros

```http
GET /api/condominios?NomeDoCondominio=Residencial&Pagina=1
```

## Cache

A API utiliza cache em memória para reduzir consultas repetidas ao banco.

O cache está aplicado na camada de Service, não no Controller.

Atualmente, o cache cobre:

- busca por ID;
- busca por nome do condomínio.

### Regras de cache

| Cenário | Usa cache? |
| --- | --- |
| Buscar por ID | Sim |
| Buscar por nome | Sim |
| Buscar por nome com variação de maiúsculas/minúsculas | Sim |
| Buscar por nome com espaços extras | Sim |
| Buscar por páginas diferentes | Usa chaves diferentes |
| Buscar por nome junto com código | Não |
| Listagem sem filtro de nome | Não |
| Erro de ID não encontrado | Não cacheia |

## Arquitetura aplicada

O endpoint foi implementado com separação de responsabilidades.

```text
Controller -> Service -> Repository -> Banco de Dados
```

### Controller

Responsável por receber a requisição HTTP e retornar a resposta ao cliente.

Não contém regra de negócio.

### Service

Responsável por:

- validar entradas;
- normalizar filtros;
- aplicar cache;
- aplicar paginação;
- converter entidades para DTOs;
- tratar dados não encontrados.

### Repository

Responsável por acessar o banco de dados usando Entity Framework Core.

Executa consultas como:

- busca por ID;
- filtro por código;
- filtro por CNPJ;
- filtro por nome;
- paginação com `Skip` e `Take`.

## Testes

O endpoint possui testes automatizados para validar os principais comportamentos.

### Tipos de teste

| Tipo | Objetivo |
| --- | --- |
| Controller | Validar retorno HTTP e integração com service |
| Service | Validar regras de negócio, paginação, erros e cache |
| Repository | Validar consultas e filtros no banco |
| Integração | Validar comportamento da API em fluxo completo |

### Testes de cache

Os testes de cache validam que:

- a segunda busca por ID retorna do cache;
- a segunda busca por nome retorna do cache;
- nomes com maiúsculas/minúsculas e espaços usam a mesma chave;
- páginas diferentes usam caches diferentes;
- nome combinado com código não usa cache;
- listagem sem nome não usa cache;
- erros não são cacheados.

Cliente envia GET /api/condominios?NomeDoCondominio=Residencial
        |
        v

        |
        v

        |
